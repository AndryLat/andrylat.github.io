    Java Core

    Теория Java Core

    Теория нужна. На этом я мог бы и остановиться, но все же стоит раскрыть этот пункт. Теория нужна и ей необходимо качественно оперировать. Это значит, что вам обязательно нужно знать весь Java Core, понимать (!) устройство Java хотя бы поверхностно и знать всю основную функциональность стандартной библиотеки. Но важно не зарываться в самые дебри, которые будут требовать много времени и не дадут вам результата. Без примеров тут не обойтись. Самая базовая вещь - это типы данных и работа с ними. Кажется все просто: int занимает 4 байта, а byte - занимает 1 байт, то есть объект, который объявлен так:

    class SomeObject {
  int a;
  byte b;
}

    Сначала можно подумать, что в памяти он будет занимать (4+1) байт, но это не так: есть еще заголовки объекта, которые “весят” 12 байт. Хорошо, значит объект весит (12+4+1) байт? Нет, согласно Java memory alignment наш объект будет занимать 24 байт (17 будет “выравнено” до ближайшей кратности 8). Это дебри работы Java, но лучше это знать и понимать. Может оно никогда вам не пригодится в реальной жизни (не везде же пишут highload системы), но на собеседовании “блеснуть” очень даже можно, ну и вам полезно, конечно же. Все подобные мелочи интересно поданы в книге “SCJP Sun Certified Programmer for Java 6”. К сожалению она уже безвозвратно устарела, но в ней можно найти множество технических деталей, объясненные доступным английским языком.

    Какую теорию по Java Core must have изучить?

    Не стоит пытаться освоить всю теорию за один вечер или даже месяц. Тут как с английским: 7 раз в неделю по 20 минут дадут значительно больше результата, чем 7 часов подряд. Банально можно загуглить “вопросы на собеседовании java junior”, выбрать себе список и осваивать его. Нельзя заучивать ответы, их нужно понимать и разбирать, идеально, если вы будете тестировать какие-то вещи с помощью кода.

    Итак:

    • Типы данных и как Java с ними работает. Мой любимый пример: почему так? (см. код ниже)
      byte b = 3;
  
  b += 7; // <-- OK
  
  b = b + 7; // <-- Compilation error!

    • Управляющие конструкции. Тут без комментариев.

    • Классы, объекты и все, что с этим связано. Нужно понимать, какие бывают модификаторы, чем отличается интерфейс от абстрактного класса, что такое override и overload методов, какие бывают вложенные классы, ну и все такие базовые вещи.

    • Коллекции Важно понимать как они устроены внутри, чтобы легко выбирать для каждой конкретной ситуации подходящую. Выбрав некорректную коллекцию для какой-то специфической задачи, можно обрушить производительность операции. Я всегда задаю следующую задачу (удивительно, но редко кто отвечает быстро и верно):

      Мы знаем, что сначала в нашу систему придет запрос на добавление числа в любое место в списке 5 миллионов раз. Потом будут приходить запросы на получение значения из любого места в списке (вставки больше не будет). Какую коллекцию лучше использовать: LinkedList или ArrayList?

      Реализация класса может быть переписана, главное соблюдать интерфейс.

      interface Storage {
  void insert(int position, Integer value);
  Integer get(int position);
}

class StorageImpl implements Storage {
  private List<Integer> storage = ?;

  public void insert(int position, Integer value) {
      storage.add(position, value);
  }

  public Integer get(int position) {
      return storage.get(position);
  }
}

    • Обработка ошибок Очень частая проблема с этой частью. Тема действительно сложная, так как за простой конструкцией try-catch-finally скрывается один из важнейших элементов работы: обработка ошибок, которые возникают во время работы приложения. Любое enterprise приложение должно писаться так, чтобы оно было максимально отказоустойчивым: даже если где-то что-то пошло не так, приложение не должно умирать с кодом -1, оно должно функционировать и извещать пользователя о проблеме, либо же уметь выходить из сложившейся ситуации.

      Понять тонкости и научиться обрабатывать ошибки можно только на практике. Я это покрою отдельной темой.

      А сейчас запомните, как делать нельзя.

    Пример 1:

      ...
  public static void main(String args...) throws IOException, ParseException {
    ...
  }
  ...

    Вопрос: кто будет обрабатывать выбрасываемые эксепшены, если выше метода main() только JVM ?

    Пример 2:

      ...
  public void someMethod()  {
    try {
        methodWhichThrowsException();
    } catch (Exception ex) {
        ex.printStacktrace();
    }
  }
  ...

    Вопрос: если у вас вебсервис, как вызывающая система узнает об ошибке, если вы вывели ее в консоль себе на сервере? Такой код очень опасен, так как эту ошибку можно долго не замечать, а потом придется исправлять много-много данных, которые стали некорректными из-за баги. Замечание: Ловить просто “Exception” очень плохо, так как мы делаем “заглушку” на любую ошибку, которая может возникнуть, а это неправильно. Exceptions используются для того, чтобы обработать словить и корректно обработать ошибку, о которой программист позаботился.

    Пример 3:

      ...
  public void someMethod()  {
    try {
        methodWhichThrowsException();
    } catch (Exception ex) {
        throw new Exception("some error was occured");
    }
  }
  ...

    Замечание: это, наверное, самое худшее, что вы можете сделать для саппорт команды – проглотить изначальный эксепшн, в котором скрывается основная информация и стектрейс что именно произошло, в каком месте и по какой причине. В логах же будет значится только многозначительное “some error was occured”, которое никому ни о чем не скажет. Не делайте так, пожалуйста.
